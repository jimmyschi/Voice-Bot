# Architecture

## How the system works

The bot is built around Twilio's Media Streams API, which lets us intercept the raw audio of a phone call in real time over a WebSocket.  When `main.py` dials the Athena test number, Twilio answers the call and immediately POSTs to our `/twiml` endpoint asking for instructions; we respond with a `<Connect><Stream>` TwiML directive that opens a second, persistent WebSocket connection back to our server at `/media-stream`.  From that point on, every 20 ms chunk of µ-law audio that the Athena agent produces arrives on the WebSocket as a base64-encoded payload.  Our `AudioPipeline` class forwards those raw bytes directly to Deepgram's streaming speech-to-text API (which accepts µ-law 8 kHz natively, so no transcoding is needed) and waits for Deepgram's endpointing signal — a 600 ms silence window that indicates the agent has finished a complete utterance.  Once a final transcript lands, it goes into a asyncio queue that a background task is draining.  That task calls the `PatientAgent`, which holds a rolling conversation history and calls the Anthropic API with a system prompt that locks the model into a specific patient persona and goal for the current scenario.  The response text is then synthesized by Deepgram's Aura TTS API (also in µ-law 8 kHz, so again no re-encoding), chunked into 160-byte frames, and streamed back to Twilio over the same WebSocket, where it plays to the Athena agent as the patient's voice.  Twilio records the mixed audio of both sides and, when the call ends, POSTs the recording URL to our `/recording-status` callback; we download the MP3 immediately so it is available alongside the transcript.  After all calls complete, a separate offline analysis pass sends each transcript to a more capable model that acts as a QA reviewer, looking for scheduling errors, factual mistakes, missed urgency escalations, and conversational quality issues, and outputs the findings as a structured Markdown bug report.

## Key design decisions

**Deepgram for both STT and TTS.**  Choosing one provider for the full audio path eliminates a transcoding step on each turn: Twilio sends µ-law, Deepgram STT consumes µ-law, Deepgram TTS produces µ-law, and Twilio plays µ-law.  Everything stays at 8 kHz mono with no lossy resampling.  An alternative like ElevenLabs produces higher-fidelity voice but requires converting from 22 kHz PCM to 8 kHz µ-law via a library like pydub, which adds latency and a dependency; for a test bot the marginal voice quality gain is not worth the complexity.

**Async throughout, with a dedicated response-processor task per call.**  The audio-forwarding path (called ~50 times per second) and the LLM+TTS path (1–3 seconds per turn) operate at very different timescales.  Running them on the same coroutine would either block audio forwarding during inference or require complex buffering.  Instead, `AudioPipeline.start()` spawns a single background `asyncio.Task` that drains a transcript queue; the hot path just enqueues final transcripts and returns immediately.  This keeps the event loop responsive and makes the latency of one turn completely independent of the audio-forwarding rate.

**Patient persona runs on the faster Haiku model; bug analysis runs on Sonnet.**  During live calls, every extra second of LLM latency is dead air on the phone.  Haiku produces a coherent 1–3 sentence patient reply in under a second.  Bug analysis is an offline batch job where accuracy matters more than speed, so we use the larger Sonnet model there.

**Scenarios are data, not code.**  All 17 test cases are plain Python dataclasses in `bot/scenarios.py` with no imperative logic.  Adding a new scenario is a one-liner dict; the entire calling, transcription, and analysis pipeline is scenario-agnostic.  This made it easy to iterate on scenario design without touching any call-handling code.
